# План реализации `POST /ai/get_info`

## 1. Цель

Реализовать первый AI-эндпойнт из [`docs/ai_endpoints.txt`](./ai_endpoints.txt), который:

- принимает сырой текст исторического документа;
- сам определяет, относится ли документ к типу `single` или `plural`;
- возвращает строго валидный JSON по контракту;
- для `single` извлекает полную карточку человека и локализует её на `ky`, `ru`, `en`, `tr`;
- для `plural` возвращает только список `normalized_names`;
- не сохраняет ничего в индекс и не пишет в БД.

Сейчас реализуется только `POST /ai/get_info`.


## 2. Что считаем успешной реализацией

Эндпойнт считается реализованным корректно, если:

- на валидный вход он всегда возвращает JSON, соответствующий одной из двух форм контракта;
- наружу никогда не уходит "почти JSON", текст вместо JSON или сломанная структура;
- `type` всегда присутствует и принимает только `single` или `plural`;
- для `single` всегда присутствуют 4 языковых блока;
- для `single` отсутствующие поля возвращаются как `null`;
- для `single` `missing_fields` и `warnings` формируются стабильно;
- для `single` `normalized_name` вычисляется детерминированно на стороне сервиса;
- для `plural` `normalized_names` возвращаются как список без дублей;
- если в `plural`-документе нет имён, сервис возвращает пустой список `[]`;
- любой успешный ответ проходит серверную Pydantic-валидацию до отправки клиенту.

Важно: под "гарантированно правильный JSON" здесь понимается гарантия на уровне сервиса, а не на уровне сырого текста от LLM. Если модель не вернула структуру, удовлетворяющую схеме, сервис не отдаёт такой ответ как успешный `200`.


## 3. Почему реализация будет через шаблонизатор

Для этого эндпойнта лучше использовать шаблонизатор промптов, а не собирать строки вручную в коде.

Причины:

- промпты будут читабельными и поддерживаемыми;
- классификация и extraction будут разделены на независимые сценарии;
- можно отдельно править инструкции без переписывания бизнес-логики;
- проще добавлять few-shot примеры для `single` и `plural`;
- проще удерживать стабильность между классификацией и извлечением.

Планируемый шаблонизатор: `Jinja2`.

Режим работы:

- шаблоны хранятся в файлах;
- рендерятся с `StrictUndefined`, чтобы не терять переменные молча;
- системный и пользовательский промпт разделены;
- схема ответа задаётся не текстом в промпте, а Pydantic-моделью через Structured Outputs.


## 4. Общая идея реализации

Эндпойнт будет работать в два шага:

1. Классификация документа как `single` или `plural`.
2. Ветвление в нужный extraction-сценарий.

Почему именно так:

- контракт теперь ветвящийся, а не единый;
- для `single` и `plural` нужны разные инструкции и разные схемы ответа;
- так проще добиться стабильного JSON, чем пытаться в одном запросе решить всё сразу;
- риск ложной "карточки человека" для многосубъектного документа становится ниже.

Итоговый пайплайн:

`raw text -> validate -> classify -> branch(single|plural) -> structured extraction -> deterministic post-processing -> response validation -> client`


## 5. Почему JSON будет надёжным

JSON будет обеспечиваться несколькими слоями защиты:

1. OpenAI Structured Outputs.
2. Pydantic-схемы в коде.
3. Автоматический parse в SDK.
4. Серверная пост-валидация.
5. Детерминированная нормализация после LLM.
6. Разделение `single` и `plural` на разные схемы.

Если модель:

- отказалась;
- вернула усечённый ответ;
- не уложилась в схему;
- вернула несовместимые типы;
- выдала не ту ветку структуры,

то сервис не возвращает такой ответ как успешный `200`.

## 7. Контракт эндпойнта

### Request

```json
{
  "text": "Байтемиров Асан Жумабекович, 1899 года рождения..."
}
```

### Response для `single`

```json
{
  "type": "single",
  "result": {
    "ky": { "...": "..." },
    "ru": { "...": "..." },
    "en": { "...": "..." },
    "tr": { "...": "..." }
  },
  "missing_fields": ["death_place"],
  "warnings": []
}
```

### Response для `plural`

```json
{
  "type": "plural",
  "normalized_names": [
    "сыдыкова бурул токтогуловна",
    "маматов эсенгул кадырович"
  ],
  "warnings": []
}
```


## 8. Схемы данных

### 8.1. Входная схема API

```python
class GetInfoRequest(BaseModel):
    text: str
```

Валидация:

- `text` обязателен;
- пустая строка после `strip()` не допускается;
- слишком короткий текст вида 1-2 слова отклоняется как `422 validation_error`, если в нём недостаточно сигнала ни для `single`, ни для `plural`.


### 8.2. Схема результата классификации

```python
class GetInfoClassification(BaseModel):
    type: Literal["single", "plural"]
    warnings: list[str] = []
```

Это внутренняя схема первого шага.


### 8.3. Схема одного языкового блока для `single`

```python
class PersonCardLocalized(BaseModel):
    full_name: str | None
    normalized_name: str | None
    birth_year: int | None
    death_year: int | None
    birth_date: str | None
    death_date: str | None
    birth_place: str | None
    death_place: str | None
    region: str | None
    district: str | None
    occupation: str | None
    charge: str | None
    arrest_date: str | None
    sentence: str | None
    sentence_date: str | None
    rehabilitation_date: str | None
    biography: str | None
```


### 8.4. Внутренняя схема extraction для `single`

```python
class GetInfoSingleLLMOutput(BaseModel):
    ky: PersonCardLocalized
    ru: PersonCardLocalized
    en: PersonCardLocalized
    tr: PersonCardLocalized
    warnings: list[str]
```


### 8.5. Внутренняя схема extraction для `plural`

```python
class GetInfoPluralLLMOutput(BaseModel):
    names: list[str]
    warnings: list[str]
```

Здесь `names` могут быть сырыми именами, а финальные `normalized_names` сервис посчитает сам.


### 8.6. Внешняя схема ответа API для `single`

```python
class GetInfoSingleResponse(BaseModel):
    type: Literal["single"]
    result: dict[str, PersonCardLocalized]
    missing_fields: list[str]
    warnings: list[str]
```


### 8.7. Внешняя схема ответа API для `plural`

```python
class GetInfoPluralResponse(BaseModel):
    type: Literal["plural"]
    normalized_names: list[str]
    warnings: list[str]
```


### 8.8. Итоговая схема ответа

```python
GetInfoResponse = GetInfoSingleResponse | GetInfoPluralResponse
```


## 9. Принципиальное разделение: что делает LLM, а что делает сервис

### LLM делает

- классификацию документа как `single` или `plural`;
- для `single` извлечение смысловых полей из сырого текста;
- для `single` локализацию человекочитаемых строк для `ky`, `ru`, `en`, `tr`;
- для `single` краткую биографическую сводку;
- для `plural` извлечение имён людей, если они есть;
- генерацию пояснений в `warnings`, если в тексте есть неоднозначность.

### Сервис делает детерминированно

- валидацию входа;
- рендер промптов;
- вызов OpenAI с жёсткой схемой;
- ветвление после шага классификации;
- обработку отказов и ошибок;
- добивку отсутствующих полей до `null` для `single`;
- пересчёт `normalized_name` для `single`;
- пересчёт `normalized_names` для `plural`;
- дедупликацию имён для `plural`;
- синхронизацию числовых и датовых полей между языками для `single`;
- вычисление `missing_fields` для `single`;
- финальную Pydantic-валидацию.

Это важный архитектурный принцип: модель отвечает за вероятностную семантику, а сервис за структуру, целостность и стабильность.


## 10. Шаблоны промптов

### 10.1. `classify_system.j2`

В системном шаблоне классификации будут зафиксированы правила:

- `single` выбирать только когда документ в основном про одного человека;
- `plural` выбирать для списков, сводок, отчётов, коллективных дел и событийных документов;
- упоминание родственников, следователей или свидетелей не делает документ `plural`, если основной субъект один;
- если есть сомнение между `single` и `plural`, безопаснее выбрать `plural` и добавить предупреждение.

### 10.2. `classify_user.j2`

Пользовательский шаблон классификации содержит только текст документа и краткое напоминание о правилах выбора `type`.

### 10.3. `single_system.j2`

В системном шаблоне extraction для `single` будут зафиксированы правила:

- извлекать только из переданного текста;
- не выдумывать отсутствующие факты;
- если поля нет, ставить `null`;
- числовые и датовые поля держать одинаковыми во всех языковых блоках;
- `normalized_name` не генерировать свободно;
- спорные или неуверенные места помечать в `warnings`;
- биография должна быть короткой и опираться только на текст;
- не смешивать данные родственников, свидетелей и других фигур с основной персоной.

### 10.4. `single_user.j2`

Пользовательский шаблон содержит:

- сам текст документа;
- список целевых полей;
- явное указание, что документ уже классифицирован как `single`;
- инструкцию сохранять исторические факты без смягчения и без художественной переработки.

### 10.5. `plural_system.j2`

В системном шаблоне extraction для `plural` будут зафиксированы правила:

- выделять только имена людей;
- не возвращать названия учреждений, должностей и географию вместо имён;
- не придумывать ФИО, если в тексте есть только статистика без списка лиц;
- если документ содержит таблицу или список, извлекать всех людей;
- если имён нет, возвращать пустой список.

### 10.6. `plural_user.j2`

Пользовательский шаблон содержит:

- сам текст документа;
- указание, что нужно извлечь все упомянутые имена людей;
- напоминание, что итоговую нормализацию сервис делает сам.

### 10.7. Почему не будем "просить JSON" текстом

Потому что основной контроль структуры будет не в промпте, а в structured schema.

Промпт нужен для качества классификации и extraction.
Схема нужна для надёжности формата.


## 11. OpenAI-механика

### 11.1. Выбранный подход

Для первого эндпойнта планируется использовать официальный Python SDK OpenAI и Structured Outputs с Pydantic-моделями.

Предпочтительный вариант:

- `client.responses.parse(...)`
- модель по умолчанию: `gpt-5.4`
- structured parsing через `text_format=<PydanticModel>`

### 11.2. Почему именно двухшаговый вызов

Предлагается делать два отдельных LLM-вызова, а не одну сложную union-схему:

1. Классификация `single/plural`.
2. Extraction по нужной ветке.

Причины:

- проще контролировать качество каждой стадии;
- проще дебажить промпты;
- ниже риск, что модель смешает две формы ответа;
- Pydantic-схемы становятся проще и надёжнее;
- ошибочная генерация карточки для `plural`-документа становится менее вероятной.

Цена решения:

- на один дополнительный вызов больше;
- немного выше latency;
- немного выше расход токенов.

Для `get_info` это оправдано, потому что качество и предсказуемость здесь важнее, чем минимальная стоимость.


## 12. Псевдокод пайплайна

```python
rendered_classify_system = render_template("get_info/classify_system.j2", {})
rendered_classify_user = render_template("get_info/classify_user.j2", {"text": request.text})

classification = client.responses.parse(
    model=settings.openai_model_get_info,
    input=[
        {"role": "system", "content": rendered_classify_system},
        {"role": "user", "content": rendered_classify_user},
    ],
    text_format=GetInfoClassification,
).output_parsed

if classification.type == "single":
    rendered_system = render_template("get_info/single_system.j2", {})
    rendered_user = render_template("get_info/single_user.j2", {"text": request.text})

    parsed = client.responses.parse(
        model=settings.openai_model_get_info,
        input=[
            {"role": "system", "content": rendered_system},
            {"role": "user", "content": rendered_user},
        ],
        text_format=GetInfoSingleLLMOutput,
    ).output_parsed

    payload = postprocess_single(parsed, classification.warnings)
    return GetInfoSingleResponse.model_validate(payload)

rendered_system = render_template("get_info/plural_system.j2", {})
rendered_user = render_template("get_info/plural_user.j2", {"text": request.text})

parsed = client.responses.parse(
    model=settings.openai_model_get_info,
    input=[
        {"role": "system", "content": rendered_system},
        {"role": "user", "content": rendered_user},
    ],
    text_format=GetInfoPluralLLMOutput,
).output_parsed

payload = postprocess_plural(parsed, classification.warnings)
return GetInfoPluralResponse.model_validate(payload)
```


## 13. Постобработка после LLM

Это обязательный этап. Без него endpoint будет слишком хрупким.

### 13.1. `normalized_name` для `single`

`normalized_name` должен вычисляться только в коде.

Алгоритм:

1. Берём канонический `full_name`, приоритетно из `ru`, иначе из первого непустого блока.
2. Приводим к lowercase.
3. Заменяем `ё` на `е`.
4. Удаляем лишние пробелы.
5. Удаляем лишнюю пунктуацию вокруг имени.

Пример:

`" Байтемиров   Асан Жумабекович "` -> `"байтемиров асан жумабекович"`

### 13.2. `normalized_names` для `plural`

Для `plural` сервис берёт список `names` от модели и сам строит итоговый `normalized_names`.

Правила:

- каждое имя нормализуется тем же детерминированным алгоритмом;
- пустые строки отбрасываются;
- дубли убираются;
- порядок по возможности сохраняется как в документе;
- если после очистки имён не осталось, возвращается `[]`.

### 13.3. Синхронизация инвариантов для `single`

Поля ниже являются инвариантами и после LLM приводятся к одному значению во всех языках:

- `normalized_name`
- `birth_year`
- `death_year`
- `birth_date`
- `death_date`
- `arrest_date`
- `sentence_date`
- `rehabilitation_date`

Если значения конфликтуют, сервис:

- добавляет предупреждение;
- выбирает каноническое значение по приоритету:
  - `ru`
  - `ky`
  - `en`
  - `tr`

### 13.4. Вычисление `missing_fields` для `single`

`missing_fields` строится не моделью, а сервисом.

Правило:

- если поле пустое во всех 4 языках, оно попадает в `missing_fields`;
- `normalized_name` в `missing_fields` не включаем, если смогли вычислить его детерминированно;
- `warnings` не влияют на `missing_fields`.

### 13.5. Объединение `warnings`

Финальный `warnings` складывается из:

- предупреждений классификации;
- предупреждений extraction;
- детерминированных предупреждений сервиса.

После этого список:

- очищается от пустых строк;
- дедуплицируется;
- сохраняет исходный порядок по возможности.


## 14. Правила классификации

Нужно заранее зафиксировать, как именно сервис понимает `single` и `plural`.

### 14.1. Считаем `single`, если

- в документе есть одна явно центральная персона;
- дело, справка, биография или карточка в основном посвящены одному человеку;
- упоминаются другие люди, но только как родственники, свидетели, следователи или косвенные фигуры.

Примеры:

- `delo_baytemirova.txt`
- `delo_sydykova.txt`
- `spravka_reabilitatsiya_1958.txt`

### 14.2. Считаем `plural`, если

- документ представляет собой список лиц;
- документ содержит сводную статистику или отчёт;
- это групповое дело без одной доминирующей фигуры;
- документ описывает несколько людей как равноправных субъектов.

Примеры:

- `spisok_oshskaya_1938.txt`
- `pismo_komissiya_1956.txt`
- `delo_toktogulova.txt`

### 14.3. Что делать в пограничных случаях

Если документ:

- содержит одну основную фигуру, но много второстепенных имён, выбирать `single`;
- содержит несколько равноправных фигур, выбирать `plural`;
- трудно классифицировать уверенно, выбирать `plural` и добавлять предупреждение.

Причина такого приоритета простая: ошибочно вернуть `plural` для почти-`single` менее опасно, чем придумать одну "главную" карточку там, где её нет.


## 15. Обработка неоднозначных случаев

### 15.1. В `single`-документе есть имена родственников

Это не делает документ `plural`.
Карточка строится только по основной персоне.

### 15.2. В `plural`-документе есть статистика, но нет имён

Возвращаем:

```json
{
  "type": "plural",
  "normalized_names": [],
  "warnings": []
}
```

### 15.3. В `plural`-документе встречаются только фамилии или сокращения

Модель может вернуть то, что реально удалось достать, но сервис не должен выдумывать полные ФИО.
Если имя слишком неполное для нормальной идентификации, допустимо оставить предупреждение.

### 15.4. Есть только косвенная дата

Для `single` сервис не должен придумывать точную дату из косвенных намёков.

Например:

- если известно только `1937 год`, нельзя возвращать `1937-01-01`;
- если указан только год, дата остаётся `null`.

### 15.5. Перевод не удался

Если в режиме `single` один языковой блок не удалось локализовать уверенно:

- допускается временно оставить исходное значение;
- в `warnings` добавляется пояснение;
- структура ответа при этом не ломается.


## 16. Ошибки и их поведение

### 16.1. `400 bad_request`

Когда:

- тело запроса отсутствует;
- JSON сломан;
- нет обязательного поля `text`.

### 16.2. `422 validation_error`

Когда:

- `text` пустой после `strip()`;
- текст слишком короткий и не содержит достаточного сигнала для классификации;
- финальный объект не проходит бизнес-валидацию.

### 16.3. `500 internal_error`

Когда:

- OpenAI недоступен;
- модель вернула отказ;
- structured parse не удался после допустимого количества попыток;
- возникла непредвиденная ошибка постобработки.

Формат ошибки сохраняется по контракту:

```json
{
  "error": {
    "code": "internal_error",
    "message": "failed to extract structured document info"
  }
}
```


## 17. Нужен ли retry

Да, но только ограниченный.

План:

- `1` основной запрос;
- `1` повторная попытка только при транзиентной ошибке сети, rate limit или кратковременной ошибке провайдера;
- без бесконечных повторов;
- без повторов, если вход явно плохой.

Повтор не нужен для "доламывания" структуры, потому что её обеспечивает схема. Повтор нужен только для инфраструктурной устойчивости.


## 18. Где в этой архитектуре будущие метаданные для RAG

Хотя `save_doc` сейчас не реализуется, уже на этапе `get_info` надо понимать, какие данные позже попадут в индекс.

### 18.1. Для `single`

Минимальный набор будущих метаданных:

- `person_full_name`
- `person_normalized_name`
- `person_birth_year`

Рекомендуемо дополнительно:

- `person_region`
- `person_charge`
- `person_district`

### 18.2. Для `plural`

Минимальный набор:

- `normalized_names`

При необходимости позже можно добавить:

- `detected_name_count`
- `document_scope = plural`

### 18.3. Зачем это нужно

Чтобы на этапе будущего RAG можно было делать поверхностный поиск и фильтрацию:

- по точному или нечёткому ФИО;
- по году рождения;
- по сочетанию `ФИО + год рождения`;
- по наличию конкретного имени внутри многосубъектного документа.


## 19. Тестовый план для первого эндпойнта

### Unit-тесты

1. Нормализация ФИО работает детерминированно.
2. `single`-ответ добивается `null`, а не пустыми строками.
3. `missing_fields` вычисляется сервисом корректно.
4. `plural`-имена нормализуются и дедуплицируются.
5. Ошибка OpenAI превращается в контрактную ошибку API.

### Интеграционные тесты

1. На тексте `delo_baytemirova.txt` эндпойнт определяет `type = single`.
2. На тексте `delo_sydykova.txt` эндпойнт определяет `type = single`.
3. На тексте `spisok_oshskaya_1938.txt` эндпойнт определяет `type = plural` и возвращает список имён.
4. На тексте `pismo_komissiya_1956.txt` эндпойнт определяет `type = plural`.
5. На тексте `delo_toktogulova.txt` эндпойнт не ломается и уходит в `plural`.
6. На слишком коротком тексте эндпойнт возвращает `422`.
7. На валидном тексте эндпойнт всегда отдаёт объект, проходящий Pydantic-валидацию.


## 20. Что не входит в первую реализацию

На этом этапе не реализуются:

- `POST /ai/save_doc`
- `POST /ai/chat`
- `POST /ai/voice`
- `POST /ai/asr`
- векторная индексация
- хранение метаданных в БД

Но документ специально описывает будущие metadata-поля, чтобы потом не переделывать extraction-слой.


## 21. Основные технические решения, которые предлагается утвердить

1. Backend framework: `FastAPI`.
2. LLM integration: официальный Python SDK `openai`.
3. Формат structured extraction: `responses.parse(...)` + Pydantic-модели.
4. Промпты: `Jinja2` шаблоны в отдельных файлах.
5. Классификация `single/plural` делается внутри эндпойнта.
6. Для надёжности используется двухшаговый пайплайн: сначала классификация, потом extraction.
7. `normalized_name` и `normalized_names` считаются только детерминированно в коде.
8. `missing_fields` считается только детерминированно в коде и только для `single`.
9. Будущие metadata для RAG:
   - для `single`: `full_name`, `normalized_name`, `birth_year`
   - для `plural`: `normalized_names`
