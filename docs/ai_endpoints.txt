AI Service API Contract
=======================

Scope
-----

Этот сервис отвечает только за AI-функции проекта:

1. извлечение структурированных полей человека из текста
2. сохранение документа в AI-индекс
3. RAG-ответы по документам
4. синтез речи
5. распознавание речи

Сервис не отвечает за:

- регистрацию пользователей
- JWT и авторизацию
- CRUD карточек человека в основной БД
- загрузку файлов с фронта
- хранение истории чатов в основной продуктовой БД

Эти задачи делает основной backend.

Общие правила
-------------

- Все AI-эндпойнты используют `POST`.
- Все идентификаторы приходят из основного backend и являются целыми числами (`int`).
- AI-сервис может хранить у себя технические данные:
  - текст документа
  - чанки
  - эмбеддинги
  - служебные метаданные индексации
- Основной backend передаёт в AI-сервис уже очищенный текст документа.
- Формат обмена по умолчанию: `application/json; charset=utf-8`
- Для ASR используется `multipart/form-data`
- Для TTS ответом является бинарное аудио с `Content-Type: audio/mpeg`

Рекомендуемые модели
--------------------

- Extraction / chat: `gpt-5.4`
- Embeddings: `text-embedding-3-large`
- TTS: `tts-1-hd`
- ASR: `gpt-4o-transcribe`

Режим выбора:

- Сейчас для демо приоритетом является качество, а не стоимость.
- Поэтому используются максимально сильные модели из доступных для данного сценария.
- Если позже понадобится удешевление или ускорение, можно перейти на:
  - `gpt-5.4-mini` для extraction / chat
  - `text-embedding-3-small` для embeddings
  - `gpt-4o-mini-tts` для TTS


1. POST /ai/get_info
--------------------

Назначение:
Проанализировать сырой текст документа, определить его тип и вернуть структурированный результат.

Внутри эндпойнта сервис обязан сам классифицировать документ в один из двух типов:

- `single` — документ в основном описывает одного человека;
- `plural` — документ описывает несколько людей, список лиц, сводку, событие или иной многосубъектный контекст.

Если документ относится к типу `single`, сервис извлекает поля карточки человека и сразу подготавливает локализованные версии данных на:

- `ky`
- `ru`
- `en`
- `tr`

Если документ относится к типу `plural`, сервис не пытается строить одну карточку человека и возвращает только список `normalized_names`, найденных в тексте.

Request body:

```json
{
  "text": "Байтемиров Асан Жумабекович, 1899 года рождения..."
}
```

Примечания:

- Эндпойнт сам определяет `type` на основе содержимого документа. Клиент не передаёт `type` в request.
- `type` обязателен в каждом успешном ответе и может принимать только:
  - `single`
  - `plural`
- Для `single` эндпойнт отвечает и за extraction, и за перевод/локализацию данных карточки.
- Для `single` в ответе всегда должны возвращаться 4 языковых блока:
  - `ky`
  - `ru`
  - `en`
  - `tr`
- Для `single` `normalized_name` должен вычисляться детерминированно внутри сервиса после извлечения, а не как свободный текст модели.
- Для `single` `normalized_name` является техническим полем для поиска и дедупликации, поэтому не обязан переводиться между языками.
- Для `single`, если поле не найдено, возвращается `null`.
- Для `single` числовые и датовые поля должны быть одинаковыми во всех языковых блоках.
- Для `plural` возвращается только список `normalized_names`.
- Для `plural` список должен содержать только детерминированно нормализованные ФИО без дублей.
- Для `plural`, если в документе нет имён людей, возвращается пустой список:

```json
{
  "type": "plural",
  "normalized_names": [],
  "warnings": []
}
```

- Если сервис не уверен в классификации документа или в каком-либо извлечённом значении, он всё равно может вернуть результат, но обязан добавить пояснение в `warnings`.
- Если перевод на один из языков в режиме `single` не удался, сервис может временно вернуть значение на исходном языке, но обязан добавить пояснение в `warnings`.
- Этот эндпойнт ничего не сохраняет в индекс и не пишет в основную БД.


2. POST /ai/save_doc
--------------------

Назначение:
Сохранить текст документа в локальный AI-индекс, нарезать на чанки и посчитать эмбеддинги.

Request body:

```json
{
  "person_id": 12,
  "document_id": 18,
  "filename": "delo_baytemirova.txt",
  "text": "Полный текст документа...",
  “link”: ”ссылка”
}
```


Response body:

```json
{
  "status": "ok"
}
```

Примечания:

- Этот эндпойнт нужен только для AI-индекса.
- Основной backend должен сам решать, где хранится оригинальный файл.
- AI-сервис сохраняет у себя:
  - текст документа
  - чанки
  - embeddings
  - техническую метаинформацию индексации
- Если `document_id` уже существует, поведение должно быть одним из двух:
  - переиндексация документа
  - ошибка `409 conflict`

Рекомендуемое решение:

- Для MVP лучше поддержать переиндексацию по тому же `document_id`.


3. POST /ai/chat
----------------

Назначение:
Получить ответ на вопрос по уже сохранённым документам через RAG.

Request body:

```json
{
  "question": "За что его арестовали?",
  "history": [
    {
      "question": "...",
      "answer": "..."
    },
    {
      "question": "...",
      "answer": "..."
    }
  ]
}
```

Response body:

```json
{
  "answer": "Байтемиров был арестован по обвинению в контрреволюционной агитации...",
  "sources": [
    {
      "document_id": 18,
      "quote_text": "Арестован по обвинению в контрреволюционной агитации..."
    }
  ]
}
```

Примечания:

- `history` передаётся вызывающей стороной, AI-сервис не обязан хранить историю сам.
- Поиск всегда выполняется по всем документам 
- `sources` обязательны, потому что это напрямую улучшает качество демо и соответствует ТЗ.
- Если ответа в документах нет, сервис должен явно сказать, что в доступном контексте недостаточно данных.

Минимальное поведение:

- взять embedding вопроса
- найти top-k чанков
- собрать prompt
- вернуть ответ только на основе найденных документов


4. POST /ai/voice
-----------------

Назначение:
Получить текст, перефразировать в приятный текст и перевести данный текст в аудио.

Request body:

```json
{
  "text": "Ислам, 19 января, место рождения - Бишкек",
  "language": "ru"
}
```

Response:

- binary audio
- `Content-Type: audio/mpeg`

Примечания:

- Этот эндпойнт занимается саммаризацией текста и TTS.
- Язык озвучивания передаётся вызывающей стороной в поле `language`.
- Перед озвучкой текст должен быть приведён к приятной, связной форме средней длины.
- Нельзя искажать факты, даты, имена, места, должности, обвинения и другие важные исторические данные.
- Если входной текст уже достаточно хороший, допускается только мягкая стилистическая правка без изменения смысла.


5. POST /ai/asr
---------------

Назначение:
Распознать речь из аудиофайла и вернуть текст.

Request:

- `multipart/form-data`
- поле `file`: аудиофайл

Пример ответа:

```json
{
  "text": "Ассаламу алейкум, Жомарт! Кандайсың?"
}
```

Примечания:


- Возвращаем только распознанный текст.


Ошибки
------

Единый формат ошибок:

```json
{
  "error": {
    "code": "bad_request",
    "message": "required field is missing"
  }
}
```

Рекомендуемые коды:

- `400 bad_request`
- `404 not_found`
- `409 conflict`
- `422 validation_error`
- `500 internal_error`


Что нужно сделать в реализации
------------------------------

MVP-порядок:

1. `POST /ai/get_info`
2. `POST /ai/save_doc`
3. `POST /ai/chat`
4. `POST /ai/voice`
5. `POST /ai/asr`

Почему так:

- первые три эндпойнта закрывают основную ценность хакатона
- `voice` и `asr` полезны, но не критичны для MVP


Интеграционный комментарий для backend-команды
----------------------------------------------

Основной backend должен:

- хранить продуктовые сущности у себя
- передавать в AI-сервис `person_id`, `document_id` и текст документа
- сам хранить историю комнат и сообщений, если это продуктовая логика
- использовать ответ `sources` для отображения ссылок на источники в UI

AI-сервис должен:

- индексировать документы
- искать релевантные чанки
- генерировать ответ
- делать TTS
- делать ASR
